# Playback Ad Blocking Design

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Reduce playback-time ads across HLS streams and embedded playback surfaces while preserving playback stability as the primary constraint.

**Architecture:** Use a layered defense inside the playback webview. First, the Rust backend filters high-confidence HLS ad segments before hls.js sees them. Second, the player page applies high-confidence request handling for known ad resources that appear during playback. Third, the playback page performs targeted DOM cleanup for obvious overlays, banners, and full-screen ad layers. Each layer must fail open so playback can continue when a rule is missing or incorrect.

**Tech Stack:** Vue 3, hls.js, Tauri 2.x, Rust, DOM APIs, Vitest

---

## Background

The app currently encounters several ad forms during playback:

- HLS stream ads embedded as playlist segments
- Top and bottom banner ads rendered in the playback page
- Floating overlays, full-screen masks, and ad iframes that sit above the video

These are not solved by a single mechanism. Segment filtering helps with stream ads, but page overlays require request blocking and DOM cleanup. A stable solution must avoid breaking normal playback controls, line switching, fullscreen, or source fallback.

## Goals

- Remove high-confidence ad segments from HLS playlists.
- Block known ad resources within the playback webview when they are easy to identify.
- Hide or remove obvious playback-page ad overlays without touching core player controls.
- Keep playback working if any filter fails.
- Make every rule easy to extend, disable, or whitelist.

## Non-Goals

- Perfectly removing every possible ad on every site.
- Breaking into site scripts to defeat all player-side ad logic.
- Aggressive, unconditional DOM stripping across all webviews.
- User-facing ad reporting or manual "skip ad" controls.

## Scope

This design covers three ad categories:

- `Stable to block`
  - HLS/m3u8 ad segments
  - Known ad domains and ad scripts
  - Known ad iframes, overlays, and banner containers
  - Obvious top/bottom ad layers in playback pages

- `Best effort`
  - Dynamically inserted full-screen ad masks
  - Countdown overlays generated by player scripts
  - Variant resources from sites that change ad hosts over time

- `Not guaranteed`
  - New or obscure ad logic that cannot be identified with high confidence
  - Player controls, subtitles, seek bars, and other core video UI
  - Embedded players whose structure is too coupled to site functionality

## Architecture

### Layer 1: HLS Segment Filtering

The Rust backend remains the authoritative place for HLS filtering. `fetch_hls_manifest` already proxies manifest requests; that path should keep rewriting relative URLs and then remove high-confidence ad segments from the returned playlist.

The filtering logic must only target playlists that actually contain segment rows. Master playlists should continue to pass through unless the resolved variant playlist is cleaned downstream. If filtering fails, the backend should return the original playlist rather than fail the request.

### Layer 2: Playback Resource Guard

The playback page should block or reroute only known ad resources:

- requests to known ad domains
- ad scripts with stable URL patterns
- obvious ad iframe sources
- obvious image/video assets that are not part of the player UI

This layer is for high-confidence matches only. It should not be a generic adblock list that can silence legitimate player assets. The safest default is to keep the rule set narrow and whitelist any resource that proves necessary for playback. The first implementation stays inside `src/views/PlayerPage.vue` and its loader/path handling rather than adding a whole-app network interceptor.

### Layer 3: Playback Page DOM Cleanup

The playback page should remove or hide ad containers only after the page has rendered enough structure to identify them. This includes:

- top and bottom banner areas
- floating overlays
- full-screen masks
- ad iframe wrappers
- countdown-based ad blocks

DOM cleanup must never delete the player stage, playback controls, episode selectors, line switchers, or fullscreen affordances. If the cleanup logic cannot distinguish an ad node from a core player node, it should do nothing.

## File Responsibilities

- `src-tauri/src/services/ad_blocker.rs`
  - HLS playlist filtering and ad-pattern matching
  - High-confidence unit tests for segment removal and pass-through behavior

- `src-tauri/src/services/resolver.rs`
  - Integrate HLS filtering into `fetch_hls_manifest_internal`
  - Preserve the existing fallback behavior when manifest fetching or filtering fails

- `src-tauri/src/services/mod.rs`
  - Re-export any service types or helper modules needed by the resolver

- `src/views/PlayerPage.vue`
  - Route HLS manifest and segment requests through the Rust proxy
  - Add a narrow request-blocking layer for known ad resources
  - Add playback-page cleanup for overlays and banner containers

- `src/components/player/*`
  - Keep playback UI unchanged unless a test proves an element is safe to hide or remove

## Data Flow

1. The player asks for a playback source.
2. If the source is HLS, the custom hls.js loader sends the manifest through `fetch_hls_manifest`.
3. The Rust backend rewrites URLs and removes high-confidence ad segments.
4. hls.js plays the cleaned manifest and continues loading segments.
5. The page-level request guard blocks obvious ad resources from loading.
6. The DOM cleanup pass removes only the ad containers that are clearly separate from playback controls.
7. Any failure in steps 3 through 6 falls back to the unmodified playback path.

## Failure Handling

- If manifest filtering fails, return the original playlist.
- If request blocking hides a resource needed by playback, add a whitelist entry for that resource and keep the rest of the rules active.
- If DOM cleanup hides a player control, stop that cleanup path for the current site or selector family.
- If a site cannot be classified safely, prefer HLS filtering only and skip aggressive DOM stripping.
- Every blocked resource or removed node should be traceable in logs by rule name or selector name.

## Testing Strategy

### Rust tests

- Verify ad segments are removed from a sample playlist.
- Verify clean playlists are unchanged.
- Verify master playlists pass through unchanged.
- Verify empty input returns empty output.
- Verify discontinuity markers are not left in broken positions after an ad segment is removed.

### Frontend tests

- Verify the custom hls.js loader routes `.m3u8` requests through the manifest proxy.
- Verify non-HLS resources still use normal loading unless a high-confidence ad rule matches.
- Verify request blocking does not affect the player container or controls.

### Manual validation

- Play at least one HLS source known to contain ad segments.
- Confirm the stream still starts and continues after filtering.
- Confirm controls, fullscreen, seeking, and line switching still work.
- Check a page with overlays to make sure the ad layers are hidden without breaking playback.

## Rollout Notes

- Ship the HLS filter first, because it has the lowest risk and the highest stability gain.
- Add resource blocking second, keeping the initial rule set narrow.
- Add DOM cleanup last, and only expand it after playback regression checks pass.
- Keep a whitelist ready for any resource or selector that proves necessary for normal playback.

## Initial Policy

- Scope playback cleanup to `src/views/PlayerPage.vue` first, because that is where the app already centralizes playback state, hls.js loading, and fullscreen handling.
- Treat request blocking as playback-session scoped unless a future task proves a broader hook is safe.
- Default to a denylist that only contains high-confidence ad hosts, ad paths, and ad iframe sources observed in real playback.
- Prefer skipping a rule over risking removal of a player control or a required media asset.
